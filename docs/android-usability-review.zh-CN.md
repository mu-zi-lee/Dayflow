# Dayflow Android 可用性评审与产品优化方案

日期：2026-07-18  
范围：`android/` 独立采集端  
版本：1.0（推荐决策已收敛）  
性质：只读代码评审与产品方案，不包含代码修改

## 1. 结论

Android 端已经具备一条完整的技术链路：定时截屏、前台应用识别、敏感应用排除、本地队列、扫码配对、局域网加密同步、开机恢复、通知栏和快捷设置控制。它目前更像一个面向开发者的“采集器控制台”，还不是普通用户可以放心安装后长期不管的产品。

产品定位确定为：**Android 首版是隐私优先、依赖 Mac 的无感采集伴侣，不是独立时间线客户端。** 它负责可靠采集、隐私过滤、离线暂存和同步；浏览、编辑、分析与长期存储继续由 Mac 端负责。

下一阶段最重要的目标不应是增加功能，而是让用户随时能回答四个问题：

1. 现在是否真的在记录？
2. 哪些内容不会被记录？
3. 数据是否已经安全到达 Mac？
4. 出问题时，我下一步该做什么？

建议把产品体验从“所有设置平铺在一个页面”重构为“首次设置向导 + 日常状态首页 + 独立设置/诊断页”。在此之前，应先修复五项 P0：队列暂停无法自动恢复、已同步图片不能保证按承诺清理、相机权限时机错误、同步结果不可见，以及前台应用身份不可信时仍可能截图。

首版明确不做：Android 本地时间线、AI 分析、活动编辑、账号云同步、多台 Mac 管理和高级采集参数。先把单台 Android 与单台 Mac 的闭环做可靠。

## 2. 当前用户旅程

从代码推断，当前路径如下：

1. 首次打开应用，系统立即同时请求通知与相机权限。
2. 首页展示录制状态、待同步图片、Mac 配对、四项设备权限、排除应用设置。
3. 用户点击“开始录制”；若无障碍服务未开启，直接进入系统无障碍设置。
4. 返回应用后，服务每 10 秒截屏一次；息屏自动暂停，解锁后恢复。
5. 用户扫描 Mac 配对码；WorkManager 在后台发现 Mac 并同步。
6. 同步成功的数据被标记为已确认，代码声明在 24 小时后清理本地文件。

技术流程闭环已经存在，但用户体验缺少“引导、反馈、恢复”三个闭环。

### 2.1 现有代码基线与约束

本方案不是重新设计一套 Android 架构，而是沿用现有模块边界逐步补齐状态和交互：

| 现有模块 | 已有能力 | 直接影响的优化判断 |
| --- | --- | --- |
| `MainActivity.kt` | 单页 Compose 控制台、权限入口、扫码入口、图片预览 | 可以先在同一 Activity 内拆分“设置向导/首页/诊断”状态，不需要为了改界面先引入多 Activity 架构 |
| `MainViewModel.kt` | 聚合录制、队列、配对和排除应用状态 | 继续作为 UI 聚合层，但同步状态、最近成功采集时间不能再只存在于进程内存 |
| `CaptureService.kt` | 10 秒循环、暂停原因、前台服务通知、息屏恢复 | 保留现有采集循环；补充权限/配对/容量条件检查和真实健康状态，不重写截屏机制 |
| `ContinuousCaptureAccessibilityService.kt` | 通过 Android 无障碍 API 截屏并负责服务恢复 | 这是现有方案的核心系统能力；首版继续限定 Android 15+，优先验证稳定性和政策风险 |
| `ForegroundAppReader.kt` | 通过 UsageStats 获取前台应用，使用缓存兜底 | 权限失效时可能继续返回旧应用，隐私判断不能把缓存结果当作当前可信结果 |
| `CapturePauseState.kt` | 组合 `USER`、`SCREEN_OFF`、`QUEUE_FULL` 三种暂停原因 | 适合扩展为统一条件状态机；容量、权限和配对状态必须支持条件消失后自动解除 |
| Room `captures` 表 | 持久化图片元数据、同步状态、确认和删除时间 | 继续作为可靠队列；若增加健康/同步历史字段，必须提供 Room migration，不能破坏未同步数据 |
| `SyncWorker.kt` | Mac 发现、注册、manifest 去重、逐项上传、WorkManager 重试 | 保留协议和队列模型；增加持久化进度/错误，并在确认后执行可靠文件删除 |
| `PairingStore.kt` | 配对信息由 Android Keystore 加密保存 | 安全基础可继续使用；解除配对前需要先处理仍在队列中的数据 |
| `PrivacyPreferences` | 包名/应用名关键词匹配和用户排除集合 | 先保留现有机制，但 UI 必须区分“系统保护规则”和“用户精确排除”，不能继续用绝对保护文案 |
| `BootReceiver`、通知、快捷设置磁贴 | 开机恢复、暂停/继续/停止快捷控制 | 应继续保留，但所有入口必须读取同一套持久化状态，避免首页、通知和磁贴互相矛盾 |

因此，推荐方案的主体是“补齐持久化状态 + 收敛单页信息 + 强化现有服务条件检查”，而不是替换 Room、WorkManager、无障碍服务或同步协议。

## 3. 主要问题与建议

### P0：发布前必须解决

#### 3.1 队列达到上限后不会自动恢复录制

证据：`CaptureService` 在待同步数据达到 10 GB 时加入 `QUEUE_FULL` 暂停原因；当前代码没有在同步释放空间后清除该原因。解锁只清除 `SCREEN_OFF`，手动继续只清除 `USER`。服务继续存活时，用户即使完成同步也可能一直看到暂停，除非停止并重新启动服务。

用户影响：用户以为同步后会自动继续，实际上可能丢失此后的时间线，而且没有清晰恢复动作。

建议：

- 将容量状态改成可恢复条件，而不是一次性锁存状态。
- 每次同步完成、定期循环和用户点击恢复时重新计算容量。
- 首页明确显示“因存储空间暂停”，并提供唯一、可执行的恢复路径。
- 不建议把 10 GB 作为唯一阈值；同时考虑设备剩余空间，预留系统安全余量。

验收标准：队列从超限降到阈值以下后，录制在不重启应用/服务的情况下自动恢复；若不能恢复，界面准确说明原因。

#### 3.2 “同步后 24 小时自动删除”的承诺与实现不一致

证据：同步成功时只写入 `deleteAfterUTCMS`；实际删除仅在后续 `SyncWorker` 执行时调用 `cleanup()`。没有一个按 24 小时到期时间调度的清理任务。若之后不再同步、Mac 不在线或用户解除配对，文件可能长期保留。

用户影响：这是隐私承诺，不只是存储优化。界面当前文字会给出比实现更强的保证。

建议：

- 将 Mac 的成功响应定义为“已经耐久写入”，收到确认后立即删除对应本地图片。
- 删除失败不能撤销 Mac 确认；应保留可重试的清理标记，并由应用/服务启动和 WorkManager 后续任务幂等重试。
- 首版不增加保留时长设置，避免扩大状态组合；只提供“删除未同步数据”这一项明确的危险操作。
- 同步确认与文件删除的先后顺序必须保证不会误删未确认数据。

验收标准：Mac 确认后本地图片立即进入删除流程；即使进程在确认与删除之间退出，重启后也能继续清理，且永不误删未确认数据。

#### 3.3 首次启动的权限请求不符合用途时机

证据：`MainActivity.onCreate()` 立即请求通知与相机权限；但相机只用于用户稍后主动扫描配对码。若用户首次拒绝相机，点击“配对 Mac”时没有专门的权限说明、重新请求或失败状态。

用户影响：首次启动即出现两个系统弹窗，信任成本高；用户还未理解产品价值就被要求相机权限。拒绝后，扫码页可能只显示不可用的相机画面。

建议：

- 首次启动不要请求相机权限；点击“扫描配对码”后再解释并请求。
- 通知权限也放到启动持续记录的上下文中请求，先说明通知用于持续状态和暂停控制。
- 对“拒绝”“不再询问”“硬件不可用”分别给出可操作状态。
- 若配对码格式适合，提供“从剪贴板导入”或手动输入作为无相机备选。

验收标准：用户可以先浏览应用而不触发权限弹窗；拒绝相机后仍能理解原因并完成配对或选择替代方式。

#### 3.4 同步缺少用户可见的状态机

证据：页面只有“立即同步”按钮和待同步数量；`SyncWorker` 的开始、发现失败、传输进度、成功时间和错误原因只写入日志，没有暴露到界面。配对密钥校验失败时甚至会自动清除配对，但用户看不到解释。

用户影响：点击按钮后没有反馈，用户不知道是 Mac 不在线、两台设备不在同一局域网、正在重试，还是已经完成。

建议：将同步表现为明确状态：未配对、等待 Mac、连接中、正在传输（x/y）、已同步、同步失败。首页至少显示“最后成功同步时间”和待同步数量；错误信息要附带下一步操作。

验收标准：任何一次手动同步在 300 ms 内出现反馈；失败后用户无需看日志就能判断该检查 Wi-Fi、Mac、配对还是权限。

#### 3.5 前台应用识别失效时仍可能继续截图

证据：`MainActivity` 只把“使用情况访问”显示为权限行，`CaptureService` 开始和每次采集前没有将其作为硬条件检查。`ForegroundAppReader.current()` 在查询不到新事件时会返回缓存的 `lastForegroundApp`。如果权限被撤销或 UsageStats 暂时不可用，服务可能继续使用旧应用身份进行隐私判断并保存新截图。

用户影响：截图可能被错误标注，更严重的是当前敏感应用可能因为沿用上一个普通应用身份而没有被排除。

建议：

- 采集服务每个周期都要得到“当前应用身份可信/不可信”的显式结果，不能用旧缓存静默降级。
- 权限缺失、查询失败或结果过期时进入可恢复暂停，不调用截屏 API。
- `ForegroundAppReader` 的缓存只可用于短暂显示，不可作为隐私授权依据。
- 权限被恢复后自动重新验证并继续，无需用户重启服务。

验收标准：运行中撤销使用情况访问后，下一个采集周期不产生图片；首页和通知显示准确原因；重新授权后自动恢复。

### P1：显著改善首次使用和日常信任

#### 3.6 缺少首次设置向导

当前四项设备权限只是平铺列表。无障碍、使用情况访问、电池优化和通知分别跳到不同系统页面，用户很难理解顺序、必要性和授权后的效果。“开始录制”失败时直接跳无障碍设置也容易造成迷失。

建议首次设置采用逐步向导：

1. 解释记录内容、频率、本地保存和同步范围。
2. 配对 Mac，先验证数据有明确出口。
3. 选择隐私保护策略，确认系统规则和默认排除项。
4. 开启前台应用识别（使用情况访问），明确未授权或失效时会停止截图。
5. 开启持续记录（无障碍），返回后在应用内验证。
6. 在真正开始后台记录时请求通知；电池优化作为“提高可靠性”的可选步骤，不伪装为硬性权限。
7. 完成一次测试截图与测试同步，再显示“设置完成”以及暂停/停止入口。

向导必须可退出、可稍后继续；每一步都应有“为什么需要”与当前验证结果。

#### 3.7 首页信息架构偏向调试工具

当前首页依次放置状态、文件路径、最近图片、Mac、权限和隐私，所有内容权重接近。应用私有目录路径即使复制出来，普通 Android 用户和其他应用也无法直接访问，价值很低；相反，“上次成功记录时间”和“上次成功同步时间”没有出现。

建议首页只保留：

- 一个强状态区：正在记录 / 已暂停 / 需要处理，并显示最近一次成功截屏时间。
- 一个主动作：暂停或继续；“停止”放入次级菜单并二次确认其语义。
- 一个同步摘要：待同步数量、占用空间、最后成功时间、错误状态。
- 一个隐私快捷入口：当前排除应用数量与“此刻不会截图”的原因。

文件路径、图片预览、协议信息和手动同步更适合放在“数据与诊断”页。最近截图预览涉及敏感内容，建议默认折叠或由用户主动开启。

#### 3.8 暂停、停止和自动暂停的语义不够清楚

系统中实际存在手动暂停、息屏暂停、容量暂停、异常停止四种状态，但首页主要只显示统一的“已暂停/发生错误”。通知正文无论暂停原因都写“息屏暂停，解锁后自动继续”，在手动暂停或队列已满时会产生误导。

建议：

- “暂停”表示保持配置、由用户恢复；“停止”表示关闭持续记录意图，需说明开机也不会恢复。
- 自动暂停必须显示原因和是否会自动恢复。
- 通知标题、正文和动作根据主暂停原因变化。
- 记录状态中加入“最近成功采集于 xx:xx”，避免服务活着但实际截图持续失败时仍显示绿色“录制中”。

#### 3.9 隐私保护的文案强于实际能力

当前所谓“银行和钱包类应用自动关闭截图”依赖英文包名或应用名称中的关键词（如 `bank`、`wallet`）。它无法覆盖所有地区和语言，也可能误伤名称包含关键词的普通应用。用户从排除列表中看到的主要是精确包名，而默认关键词规则并不直观。

建议：

- 将文案改为“尝试自动识别”，不要暗示完整覆盖。
- 首次设置时展示可理解的默认保护类别和局限。
- 对精确包名与模糊关键词规则分开管理，避免用户删除/勾选时不清楚实际效果。
- 增加“当前应用已保护”的即时反馈，并允许从通知或首页快速排除当前应用。
- 评估 Android `FLAG_SECURE`、密码字段、系统认证界面等更可靠信号；任何自动识别仍需保守处理。

### P2：产品成熟度与维护性

#### 3.10 首版设备范围应保持 Android 15+

当前 `minSdk = 35`，即只支持 Android 15 及以上。这大幅缩小可安装设备范围。若这是为了使用无障碍截屏 API或降低适配成本，应在发布范围中明确；若目标是广泛使用，需要先验证降低最低版本所需的替代方案和厂商后台限制。

决定：首轮侧载内测维持现有 `minSdk = 35`，不在核心采集链路尚未稳定时扩展兼容面。收集目标用户系统版本与设备品牌数据后，再单独评估降低最低版本；该工作不阻断首版。

#### 3.11 商店政策与用户信任风险需要单独验证

应用使用无障碍服务截屏、`specialUse` 前台服务和忽略电池优化请求。这些能力在 Google Play 审核中敏感，仅有技术实现不代表可公开上架。无障碍用途必须透明、最小化，并与商店声明和应用内披露一致。

建议在公开发布前完成政策评审；若只做私有分发，也要提供清楚的隐私说明、采集指示、数据流说明和一键停止/清理能力。

#### 3.12 测试覆盖集中在纯数据逻辑

现有单元测试覆盖暂停原因优先级和 metadata-only 线格式；缺少对真实用户风险最大的状态转换、清理调度、权限拒绝、进程重启、WorkManager 取消/重试、服务恢复和 Compose 流程测试。

建议优先增加状态机与存储生命周期测试，再增加少量关键端到端场景。不要用大量截图测试替代行为测试。

## 4. 建议的信息架构

### 日常首页

- 录制状态、状态原因、最近成功记录时间
- 暂停/继续主按钮
- 待同步数量与最后成功同步时间
- 需要用户处理的唯一最高优先级事项
- 隐私状态快捷入口

### 首次设置

- 工作原理与数据流
- 隐私排除确认
- 必要权限逐项引导与回到应用后的验证
- Mac 配对与测试同步
- 完成状态

### 设置

- 本地未同步数据管理与危险操作
- 排除应用与保护规则
- 通知、耗电与后台可靠性
- Mac 配对管理

### 数据与诊断

- 待同步/已同步数量和空间
- 最近同步日志（用户可理解的摘要，不暴露密钥）
- 最近截图预览（默认隐藏）
- 导出诊断信息
- 清理操作

### 现有代码中的实施落点

| 目标 | 优先沿用/调整的现有代码 | 建议边界 |
| --- | --- | --- |
| 首次设置与日常首页分离 | `MainActivity`、`MainViewModel` | 首版可使用 Compose 内部页面状态；只有导航复杂度实际增加后再引入导航框架 |
| 可信录制状态 | `CaptureState`、`CapturePreferences`、`CaptureService` | 将“期望录制”和“真实健康”分开；关键时间与原因持久化，不能继续只用全局 `StateFlow` |
| 可恢复暂停 | `CapturePauseState` | 保留多原因集合模型；为权限、配对、容量增加条件原因，并为每项定义设置/清除位置 |
| 安全前台应用识别 | `ForegroundAppReader`、`CaptureService` | 返回带可信度/时间的新结果；不可信时禁止调用 `takeScreenshot()` |
| 同步进度和错误 | `SyncWorker`、Room、`MainViewModel` | Worker 写持久化状态，UI 观察状态；不要只依赖 `Log` 或临时 Compose 状态 |
| 确认后删除 | `SyncWorker`、`CaptureRepository.cleanup()`、`CaptureDao` | 复用确认字段或增加清理状态；需要迁移测试和“确认后进程退出”测试 |
| 解除配对保护 | `PairingStore.clear()`、`MainViewModel.clearPairing()` | 不再由图标单击直接清除；先检查队列并给出重新配对/删除数据选择 |
| 权限按需请求 | `MainActivity.permissionLauncher`、`QrScannerView` | 将相机请求移到扫码动作，将通知请求移到启用持续记录动作；扫码页负责拒绝态 |
| 通知与磁贴一致 | `CaptureService.notification()`、`CaptureTileService` | 两者读取同一持久化状态映射，不各自推断一套录制状态 |

这些调整会增加少量状态模型和 Room migration，但不要求改变现有 AES-GCM 配对/传输方式、Mac Discovery、截图 API 或捕获文件格式。

## 8. 产品决策：逐项自问自答

以下问题按依赖顺序作出决定。后面的答案建立在前面的答案之上，不再作为开放问题留给实现阶段。

### 决策 1：Android 是采集伴侣还是完整客户端？

**答案：首版只做 Mac 的采集伴侣。**

理由：现有架构已经围绕采集、暂存和局域网同步形成闭环，而时间线展示、AI 分析和编辑能力都在 Mac。把 Android 扩展为完整客户端会立即引入数据库双向同步、冲突处理、账号体系和跨端一致性，反而推迟核心采集体验成熟。

产品后果：首页不做时间线或日报，只回答录制、隐私、同步和异常处理。所有采集数据的最终目的地是已配对 Mac。

### 决策 2：首发通过什么方式分发？

**答案：先个人/小范围侧载内测，达到可靠性门槛后再决定是否申请 Google Play。**

理由：无障碍截屏、特殊用途前台服务和忽略电池优化都存在商店政策风险；不同厂商对后台服务的限制也需要真机数据。先侧载不是回避政策，而是避免在产品行为尚未稳定时同时承担审核成本。

产品后果：阶段 A、B 不以商店审核为目标，但所有权限说明、隐私披露和停止/清理能力仍按公开产品标准设计。进入阶段 C 前单独完成政策可行性评审。

### 决策 3：是否必须先配对 Mac 才允许持续记录？

**答案：首次设置必须先成功配对并完成一次测试同步，之后允许 Mac 离线时继续记录。**

理由：如果尚无数据出口就允许长期记录，用户很容易积累大量敏感图片并触发容量暂停。另一方面，要求 Mac 永远在线会破坏移动场景，所以“首次验证出口、之后允许离线”更合理。

产品后果：首次设置包含测试截图、隐私确认和测试同步。解除配对时自动暂停新增截图，但保留并明确展示待同步数据，用户可以重新配对原 Mac、配对新 Mac或主动删除本地数据。

### 决策 4：本地图片默认保留多久？

**答案：Mac 明确确认持久化后，Android 默认立即删除图片；未确认数据绝不自动删除。**

理由：Android 是传输端而非长期存储端，屏幕截图高度敏感。确认成功后继续保留 24 小时没有明确用户价值，却扩大隐私和磁盘占用风险。协议中的“确认”必须表示 Mac 已完成耐久写入，而不是只收到请求。

产品后果：首版不提供复杂保留时长选项。诊断需要依赖元数据和脱敏日志，而不是保留原图。“立即清理已同步内容”只作为异常兜底；未同步内容的删除必须显示数量、占用空间和不可恢复警告。

### 决策 5：没有“使用情况访问”权限时是否继续截图？

**答案：不截图，只维持可恢复的“需要权限”状态。**

理由：没有前台应用身份就无法应用排除列表，也无法可靠避开密码管理器、银行或钱包应用。为了连续性而盲拍与“隐私优先”的定位冲突。

产品后果：“使用情况访问”和无障碍截屏都属于开始记录的必要条件；通知用于透明状态和控制，也在开始前完成授权。忽略电池优化仍是推荐项而非硬阻断项。

### 决策 6：记录频率固定还是自适应？

**答案：首版保留屏幕解锁期间每 10 秒一次的固定频率，不向用户开放设置；获得真机数据后再设计自适应策略。**

理由：立刻引入基于触摸、应用切换、电量和画面变化的自适应算法，会增加难以验证的漏记分支。固定频率更容易建立正确性基线，也与现有 Mac 分析输入一致。频率本身不应成为普通用户需要理解的设置。

产品后果：内测必须测量每小时图片数、平均文件大小、CPU、内存、耗电和温升。只有在时间线质量基线稳定后，才尝试“画面未变化降频、应用切换立即采集”等自适应方案，并用相同设备做 A/B 对照。

### 决策 7：敏感应用保护偏向漏记还是完整记录？

**答案：宁可漏记，也不误拍；首版不提供降低保护强度的开关。**

理由：一次敏感信息误拍的成本远高于少量时间线缺口。关键词识别只能作为补充，不能用“自动保护银行和钱包”这样的绝对文案包装。

产品后果：系统安全界面、无法识别前台应用、用户排除应用和高置信敏感类别都只记录最少元数据，不保存截图。首页应允许快速排除当前应用，但恢复某个系统默认保护规则需要更谨慎的独立流程，首版可以不支持。

### 决策 8：是否支持 Android 15 以下设备？

**答案：首轮内测维持 Android 15+；完成核心行为验证后，再以目标用户设备数据决定是否降低最低版本。**

理由：现在降低版本会把 API 差异、后台限制和厂商适配同时带入尚未稳定的产品。先限制变量有利于验证采集与同步模型。Android 15+ 不是永久产品承诺。

产品后果：首轮至少覆盖一台 Pixel 和主力用户实际使用的其他品牌设备。若超过 20% 目标用户因系统版本无法安装，或存在明确商业需求，再立项验证较低 API 的截屏与后台恢复方案。

### 决策 9：首页是否展示最近截图？

**答案：默认不展示，移入“数据与诊断”，并要求用户主动进入后再显示。**

理由：打开采集器时展示最近屏幕内容可能造成二次隐私暴露；它也强化了调试工具感，却没有帮助用户判断服务是否可靠。

产品后果：首页以“最近成功记录时间”和健康状态代替缩略图。诊断页的预览默认折叠，退出页面即重新隐藏，不在系统最近任务缩略图中暴露内容。

### 决策 10：用户停止记录后，待同步数据怎么办？

**答案：停止只停止新增采集，既有待同步数据继续按网络条件自动同步。**

理由：用户通常把“停止记录”理解为不再产生新截图，而不是放弃已经记录的数据。如果把同步一起停止，界面必须引入第二套难以解释的队列生命周期。

产品后果：停止后的首页仍显示同步进度；用户可以单独执行“删除未同步数据”，该操作需要明确确认。若用户解除配对，自动同步停止且新增采集暂停，直到重新建立有效配对。

### 决策 11：是否需要“手动立即同步”？

**答案：保留，但降级为同步摘要中的次要动作；系统仍以自动同步为主。**

理由：手动同步对排障和即将离开局域网的场景有价值，但不能成为正常数据流依赖。按钮必须防重复触发，并展示正在进行的同一项工作，而不是让用户反复创建/替换任务。

产品后果：正常状态只显示“自动同步中/已同步”；连接失败时出现“重试”动作。诊断页可保留始终可见的“立即同步”。

### 决策 12：如何定义“正在记录”？

**答案：同时满足服务健康、必要权限有效、最近一个采集周期成功，才显示“正在记录”。**

理由：仅凭前台服务存活会产生假绿色状态。截图 API 持续失败、权限被系统撤回或前台应用识别失效时，用户需要尽快知道。

产品后果：持久化最近一次成功采集时间和失败原因。连续两个周期没有成功后进入“记录受阻”，显示原因和唯一恢复动作；单次暂时失败不立即打扰用户。

## 9. 收敛后的目标体验

### 9.1 第一次打开

用户先看到简短的数据流说明：Android 截取屏幕、先在本机私有目录暂存、通过局域网加密发送到已配对 Mac、Mac 确认后立即删除本地图片。此时不弹出任何系统权限请求。

点击“开始设置”后依次完成：配对 Mac、确认默认隐私保护、开启使用情况访问、开启无障碍截屏、允许通知、可选的电池优化设置、测试截图与测试同步。每次从系统设置返回后自动验证结果，失败时停留在当前步骤，不让用户猜下一步。

### 9.2 日常打开

首屏首先显示一句可信结论，例如：

- “正在记录 · 最近成功于 14:32:10”
- “已手动暂停 · 点击继续”
- “记录受阻 · 需要重新开启使用情况访问”
- “Mac 离线 · 正在本机暂存 126 项（284 MB）”

主按钮只有“暂停”或“继续”。同步和隐私以摘要行呈现；设置、诊断和停止位于次级位置。正常状态下无需用户操作。

### 9.3 异常恢复

同一时刻只突出一个最高优先级问题，并提供一个主要动作。优先级为：隐私保护失效、存储空间不足、必要权限失效、采集服务异常、Mac/网络不可达。低优先级同步问题不能遮盖高优先级采集或隐私问题。

所有自动暂停状态都说明“是否会自动恢复”。恢复成功后保留一条短暂结果反馈，并更新最近成功采集/同步时间。

## 10. 首版完成标准

满足以下条件后，才认为 Android 采集伴侣达到可扩大小范围内测的标准：

1. 新用户无需外部说明即可完成配对、授权、测试截图和测试同步。
2. 首次打开不主动弹权限；每项权限只在对应动作上下文中请求。
3. 未配对 Mac、无法识别前台应用或必要权限失效时绝不保存截图。
4. 首页状态与最近一次真实采集结果一致，不以服务存活冒充成功记录。
5. Mac 离线后数据可暂存；恢复在线后自动续传，无重复、无静默丢失。
6. 达到容量阈值会暂停；空间释放后无需重启即可自动恢复。
7. Mac 确认持久化后，本地图片能够可靠删除；删除失败会自动重试并可诊断。
8. 手动暂停、停止、息屏暂停、容量暂停和权限异常在应用与通知中的语义一致。
9. 默认界面、最近任务缩略图和通知不暴露截图内容或敏感应用名称。
10. Pixel 与至少一个目标用户常用品牌设备连续运行 72 小时，无静默停止；耗电和磁盘增长有可接受的实测基线。
11. 核心状态机、清理、同步重试、进程重启和权限撤回具备自动化测试。
12. 隐私说明准确描述真实行为，不使用“始终”“完全”“自动保护所有”等无法证明的绝对承诺。

达到以上标准后，再评估自适应采集频率、较低 Android 版本、多台 Mac 和公开商店分发；这些都不是首版阻断项。
